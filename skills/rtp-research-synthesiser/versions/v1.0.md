# Research Synthesiser — The Weekly Intelligence Orchestrator

## Purpose

This skill transforms raw research inputs (SuperGrok X signals + Perplexity Deep Research) into compounding intelligence across Ravi's three highest-leverage knowledge systems. It is the BRIDGE between "information collected" and "intelligence applied."

Without this skill, research accumulates but never compounds. With it, every week's signals become next week's sharper questions, updated course content, refined skills, and stronger interview answers.

## Pre-Synthesis Protocol — What to Read Before Running

Every synthesis session MUST begin by reading these files in order:

```
1. CLAUDE.md                           → Current folder structure and rules
2. MASTER_INDEX.md                     → Active project states
3. 5Knowledge/rules.md                 → Confirmed patterns (do not contradict)
4. 5Knowledge/hypotheses.md            → Patterns being watched (look for evidence)
5. 1Projects/interview-prep/CONTEXT.md → Current interview prep state
6. 1Projects/ai-fluent-course/CONTEXT.md → Current course state
7. 2Skills/SKILL-REGISTRY.md           → Current skill versions
```

This ensures the synthesiser knows what EXISTS before deciding what to ADD. Never duplicate knowledge that's already captured. Never contradict confirmed rules without explicit evidence.

## Input Collection Protocol

### Step 1: Gather SuperGrok X Research

Location: `3Research/x-research/YYYY-Wxx/`

Read ALL markdown files from the current week's folder. There should be approximately 60-66 files (8 daily schedules × 7 days + 2 bi-daily schedules × 3-4 deliveries).

For each file, extract:
- **Schedule source** (which of the 10 schedules produced this)
- **High-signal findings** (posts with production evidence, not opinions)
- **Recurring themes** (same topic appearing across multiple schedules = strong signal)
- **Contradictions** (practitioners disagreeing = nuance opportunity)
- **Anthropic-specific signals** (from Schedule 3: Claude Ecosystem + Schedule 10: Anthropic Interview Intel)

### Step 2: Gather Perplexity Deep Research

Location: `3Research/perplexity-research/` (subdirectories by Space)

Read exported markdown summaries from each of the 10 Spaces for this week. If no export exists for a Space this week, note it as "no new research" — do not fabricate.

For each Space's output, extract:
- **Verified findings** (multi-source, cited — higher confidence than X signals)
- **Research that CONFIRMS X signals** (X said it, Perplexity proved it → promote to rules.md)
- **Research that CONTRADICTS X signals** (X hyped it, Perplexity debunked it → flag)
- **Depth that X lacks** (papers, case studies, data that practitioners on X didn't share)

### Step 3: Cross-Reference

This is where the synthesis skill earns its name. Create a mental cross-reference matrix:

```
For each significant finding from EITHER source:
├── Does the other source confirm it?     → HIGH CONFIDENCE
├── Does the other source contradict it?  → INVESTIGATE
├── Is it only from X?                    → SIGNAL (unverified)
├── Is it only from Perplexity?           → RESEARCH (may lack practitioner context)
└── Does it appear in 3+ separate files?  → PATTERN (candidate for rules.md)
```

## Stream 1: Interview Prep Extraction

### Destination
`1Projects/interview-prep/`

### What to Extract

This stream feeds Ravi's Anthropic PM interview preparation. Every extraction must map to one of Anthropic's five interview rounds:

| Round | What to Feed It |
|-------|----------------|
| **Recruiter Screen** | Mission alignment evidence — why Ravi's specific experience (Honeywell enterprise AI, Perplexity Fellow, 17-module curriculum, CONTEXT framework) makes him uniquely suited. Fresh examples of Ravi's values matching Anthropic's safety-first culture. |
| **Product Sense & Framing** | Claude product intelligence — recent features, user pain points, competitive positioning. Real problems Ravi could articulate solving. Specific product decisions Anthropic made this week and how Ravi would think about them. |
| **Analytical & Case** | Data points — Claude growth signals, API usage patterns, enterprise adoption metrics, competitive market sizing. Any quantitative evidence from this week's research that could strengthen a case discussion. |
| **Cross-Functional Collaboration** | Evidence of how Anthropic PMs work with researchers and engineers. Patterns from X threads showing PM-researcher collaboration. Enterprise AI collaboration patterns from Ravi's own domain that translate. |
| **Culture & Values** | AI safety debates with nuanced positions. Ethical trade-offs discussed this week. Evidence of Ravi's alignment with Anthropic's "thoughtful, deliberate approach to AI development." |

### Extraction Format

For each interview-relevant finding, produce:

```markdown
### [Finding Title]

**Source:** [SuperGrok Schedule X / Perplexity Space Y] — [date]
**Confidence:** [HIGH: multi-source confirmed | MEDIUM: single strong source | SIGNAL: unverified but interesting]
**Interview Round:** [Recruiter | Product Sense | Analytical | Cross-Functional | Culture & Values]

**The Finding:**
[2-3 sentence summary of what was discovered]

**Interview Application:**
[How Ravi would weave this into a specific interview answer. Written as a 
brief narrative snippet — not "mention this" but the actual words he might 
say, adapted to his voice: warm but precise, rigorous but never condescending]

**Connection to Ravi's Experience:**
[Specific bridge to Honeywell, Perplexity Fellowship, AI-Fluent programme, 
or CONTEXT framework. This is what makes the answer HIS, not generic.]
```

### Destination Files

| File | What Goes There |
|------|----------------|
| `war-book/anthropic-product-knowledge.md` | Claude product intelligence, competitive positioning, user pain points |
| `war-book/anthropic-culture-signals.md` | Culture evidence, employee signals, values alignment data |
| `war-book/safety-positions.md` | AI safety debates, nuanced positions on ethical trade-offs |
| `war-book/role-intelligence.md` | PM role definition at Anthropic, hiring patterns, skills emphasis |
| `narratives/` | Specific story snippets ready to deploy in interviews |
| `CONTEXT.md` | Update the project context with this week's most important shifts |
| `CHANGELOG.md` | Log what was added and why |

### Quality Gate for Interview Prep

Before writing to any interview prep file, verify:

- [ ] Is this finding SPECIFIC enough to use in an answer? ("AI safety matters" = useless. "Anthropic's approach to Constitutional AI differs from RLHF because [specific mechanism]" = useful)
- [ ] Does it connect to Ravi's UNIQUE experience? (Generic AI knowledge = weak. "At Honeywell, I faced a similar trade-off when [specific situation]" = strong)
- [ ] Is it CURRENT? (Last 7 days preferred. Older findings need to be framed as "longitudinal observation" not "recent news")
- [ ] Would saying this in an interview make the interviewer think "this person follows our work closely"? That's the bar.

## Stream 3: Skill System Extraction

### Destination
`2Skills/` (governed by SKILL-REGISTRY.md)

### What to Extract

This stream updates Ravi's skill files — both the 40 AI PM skills AND the Claude-deployed skills (writing, presentations, branding, etc.). Skills are the most sensitive files in the system because they are DEPLOYED — changes affect every future Claude session.

### What Triggers a Skill Update

Not every finding warrants a skill update. Apply these filters:

```
PATTERN CONFIRMED (3+ occurrences across different sources this week)
  → Add to relevant skill as a new rule or heuristic

PATTERN OBSERVED (1-2 occurrences, strong evidence)
  → Add to 5Knowledge/hypotheses.md, tag with relevant skill name

NEW CAPABILITY DISCOVERED (Claude or tool feature that changes a skill's approach)
  → Flag for skill revision with specific recommendation

SKILL GAP IDENTIFIED (research reveals a need no current skill addresses)
  → Draft proposal for new skill, add to SKILL-LEARNING-LOG

EXISTING RULE CONTRADICTED (evidence against a confirmed rule)
  → Flag prominently — do NOT silently change. Demote from rules.md 
    to hypotheses.md with contradicting evidence noted
```

### Skill Update Protocol

**CRITICAL:** Skills follow versioning rules from CLAUDE.md. Before ANY skill update:

1. Check `2Skills/SKILL-REGISTRY.md` for current version
2. Copy current skill to `versions/v{current}.md`
3. Make the update to the main SKILL.md
4. Increment version in SKILL-REGISTRY.md
5. Log change in the skill's CHANGELOG

### Extraction Format

For each skill-relevant finding:

```markdown
### Skill Update Proposal: [Skill Name] v[current] → v[proposed]

**Source:** [Schedule/Space] — [date]
**Evidence Strength:** [CONFIRMED: 3+ sources | OBSERVED: 1-2 strong sources | CONTRADICTING: challenges existing rule]

**Current State:**
[What the skill currently says about this topic — quote the relevant section]

**Proposed Change:**
[Specific text to add, modify, or flag for removal. Write the exact 
new content, not a description of what should change]

**Impact Assessment:**
- Sessions affected: [Which types of Claude sessions use this skill?]
- Risk level: [LOW: additive | MEDIUM: modifies existing guidance | HIGH: contradicts established rule]
- Reversibility: [Can this be undone easily if wrong?]

**Recommendation:** [APPLY / HOLD FOR REVIEW / ADD TO HYPOTHESES]
```

### Destination Files

| Destination | What Goes There |
|-------------|----------------|
| `2Skills/[skill-folder]/SKILL.md` | Only CONFIRMED changes (after versioning protocol) |
| `2Skills/ai-pm-skills/SKILL-LEARNING-LOG.md` | All observations, even unconfirmed |
| `2Skills/ai-pm-skills/SKILL-REVISION-PLAN.md` | Proposed revisions awaiting Ravi's approval |
| `5Knowledge/rules.md` | Patterns confirmed across 3+ sources |
| `5Knowledge/hypotheses.md` | Patterns observed but not yet confirmed |

### Quality Gate for Skill Updates

- [ ] Has the versioning protocol been followed? (copy old → update → increment → log)
- [ ] Is this a CONFIRMED pattern (3+) or just an interesting observation?
- [ ] Does this change contradict any existing rule in `5Knowledge/rules.md`?
- [ ] Is the impact assessment honest? (Don't mark HIGH-risk changes as LOW)
- [ ] Would this change make the skill MORE useful or just MORE verbose?

## Rules This Skill Follows

1. **Never fabricate findings.** If a week's research is thin, say so. "This was a quiet week for Module 08" is better than padding with low-value observations.

2. **Confirmed patterns only promote to rules.md.** The threshold is 3+ independent occurrences across different sources. Not 3 tweets by the same person — 3 distinct evidence points.

3. **Contradictions are flagged, never silently resolved.** If this week's research contradicts an existing rule, the synthesiser presents both sides and asks Ravi to decide. It does not unilaterally change established knowledge.

4. **Interview prep gets SPECIFICITY.** Generic AI knowledge is worthless in an Anthropic interview. Every interview prep extraction must pass the test: "Would saying this make the interviewer think 'this person follows our work closely'?"

5. **Course content gets TEACHING TRANSFORMATION.** Raw findings are not content. Every course extraction must apply at least one of Ravi's 10 Thinking Algorithms to transform information into a teachable moment.

6. **Skill updates get VERSIONING.** No skill is ever edited without the copy-old → update → increment → log protocol. This is non-negotiable.

7. **The synthesis report is the SINGLE SOURCE OF TRUTH for this week's intelligence.** All three streams' outputs should be traceable back to the synthesis report. If it's not in the report, it wasn't synthesised.

8. **Respect the zones.** This skill READS from `3Research/` (input), WRITES to `1Projects/` (interview prep, course), PROPOSES to `2Skills/` (pending approval), and UPDATES `5Knowledge/` (confirmed patterns only). It never creates files in `3Research/` — that's SuperGrok and Perplexity's job.
