# Build or Buy

## DEPTH DECISION

**Go deep if:** You're evaluating fine-tuning ROI, designing model abstraction layers, planning to switch models in future, or committing engineering resources.

**Skim to diagnostic questions if:** You want a quick decision on build vs. buy for a specific feature.

**Skip if:** Early exploration phase where you just need to ship, or pre-PMF stage.

## GROUNDING (Before Starting)

Follow the [Universal Skill Protocol](../../../UNIVERSAL-SKILL-PROTOCOL.md):
1. Ask the Grounding Questions (Section 1) — at minimum: Who is the customer? What problem? What are we saying YES to and NO to?
2. Route depth: Executive Summary or Comprehensive Analysis?
3. Identify output format: Document, presentation, spreadsheet, or inline?

Then proceed with the skill-specific analysis below.

## THE TRAP

You will choose based on team expertise, not problem fit. The bias is **skill bias** — if you have a prompt engineer, it's a prompt problem. If you have ML researchers, it needs fine-tuning. The bias confirms itself: each approach *works*, so early wins feel like validation. Six months later the economics are wrong.

The right decision depends on five variables nobody usually measures. Get all five right and the decision is obvious. Get one wrong and you've committed to 18+ months of wrong. This framework forces all five to the surface.

## THE PROCESS

Five gates. Stop at the first "no." "Yes" at all five = BUILD. "No" anywhere = BUY.

**Gate 1: Problem Type — Deterministic Output?**

Ask: Can you write a precise output spec, or is output probabilistic?

- **Deterministic:** Classification (spam/not), extraction (dates, entities), ranking → Continue
- **Probabilistic:** Generation (write, summarize), reasoning (explain, debug) → **BUY** (API only)

Why: Fine-tuning on open-ended output fails. The general model is already optimized for generation; your custom model won't beat it enough to justify overhead.

**Gate 2: Data Availability — 1000+ Labeled Examples?**

Ask: Do you have labeled training data *in production*?

- **10k+ examples:** Continue
- **1-10k examples:** Continue (marginal; risk flag)
- **<1000 examples:** **BUY** (fine-tune only works with >1k; otherwise prompt-engineer)
- **None, but can generate:** **BUY** (cost of generation > API calls, usually)
- **Proprietary/sensitive:** Continue (privacy favors building)

**Gate 3: Latency Budget — P95 Latency?**

Ask: What's your max acceptable response time?

- **<100ms:** **BUY** (API overhead is too high)
- **100-500ms:** Continue (depends on economics)
- **>500ms:** Continue (RAG and fine-tuning both possible)

**Gate 4: Cost Tolerance — Annual Cost Per User?**

Calculate: (tokens/request) × (requests/user/year) × (token price).

- **<$0.50/user/year:** **BUY** (fine-tune overhead unjustified)
- **$0.50-5:** Continue (fine-tune ROI *possible*, not guaranteed)
- **>$5:** Continue (fine-tune easily justified)

**Gate 5: Maintenance Commitment — Long-term Team?**

Ask: Can you maintain custom models 2+ years? (Re-fine-tune, monitor drift, update pipelines, debug failures.)

- **Dedicated ML engineer + infrastructure:** BUILD (if Gates 1-4 pass)
- **Part-time ML engineer:** **BUY** (risk is too high; commitment will fail)
- **No ML engineer:** **BUY** (impossible)

## Reference: Four Approaches

| Approach | Latency | Cost | Accuracy | Data | Maintenance |
|----------|---------|------|----------|------|-------------|
| **Prompt** | 100-2000ms | $0.10-1/call | +5-15% | 0 (few-shot) | Minimal |
| **RAG** | 500-5000ms | $0.20-2/call + infra | +10-30% | Docs | Moderate |
| **Fine-tune** | 100-500ms | $50-500k + $0.01-0.1/call | +15-40% | 1k+ labeled | High |
| **API** | Vendor SLA | $0.05-2/call | General-optimized | 0 | None |

## THE TRILEMMA: BUILD VS BUY VS FINE-TUNE

The "build or buy" decision is not actually binary. It's a trilemma with four orthogonal options, each with distinct cost/benefit profiles:

**Option A: Prompt Engineering with API**
- **Approach:** Write a well-crafted prompt, call a public API (GPT-4, Claude, Llama via API).
- **Cost:** $0.01-1 per call. No infrastructure cost.
- **Timeline:** Deploy in hours. Iterate in minutes.
- **Accuracy:** +5-15% above baseline (if domain is well-represented in training data).
- **When to choose:** Cost per user <$0.50/year OR latency budget <200ms OR you're prototyping OR your domain is general (not highly specialized).
- **Maintenance:** Minimal. Update prompts quarterly based on model releases.
- **Risks:** Model dependency (vendor controls capability). Accuracy ceiling limited by general-purpose model's knowledge.

**Option B: RAG + Prompt (Retrieval Augmented Generation)**
- **Approach:** Index proprietary documents, retrieve relevant context, pass to API.
- **Cost:** $0.20-2 per call + retrieval infrastructure ($500-5000/month for embedding index + storage).
- **Timeline:** 2-4 weeks to build retrieval pipeline, index data, validate quality.
- **Accuracy:** +10-30% above Option A (because domain-specific context is injected).
- **When to choose:** You have proprietary docs/data that aren't in model training. Latency budget 500-5000ms. Cost per user $0.50-5/year.
- **Maintenance:** Moderate. Update docs, monitor retrieval quality, adjust chunking/embedding strategy.
- **Risks:** Retrieval failure (wrong documents returned) corrupts output. Sensitive data exposure if index is queried without access control.

**Option C: Fine-Tuning**
- **Approach:** Collect 1k-10k labeled examples, retrain model on your data.
- **Cost:** $50k-500k upfront + $0.01-0.1 per inference call.
- **Timeline:** 4-12 weeks (data collection, training pipeline, evaluation, retraining cycles).
- **Accuracy:** +15-40% above baseline (if data is representative and label quality is high).
- **When to choose:** You have 10k+ high-quality labeled examples. Accuracy gap is >15%. Cost per user $1-5/year justifies the infrastructure.
- **Maintenance:** High. Monitor model drift, retrain quarterly, manage training pipelines, debug model failures.
- **Risks:** Data poisoning (bad labels corrupt model). Maintenance commitment fails and model stagnates. Over-fitting to historical data.

**Option D: Buy Vertical SaaS**
- **Approach:** Use a purpose-built SaaS tool designed for your use case.
- **Cost:** Varies widely ($100-10k/month depending on vendor and volume).
- **Timeline:** Integration in 1-2 weeks.
- **Accuracy:** Vendor-optimized for your domain (often better than any DIY option).
- **When to choose:** Your use case is common enough that a vendor exists. You want zero maintenance. Vendor lock-in is acceptable.
- **Maintenance:** None. Vendor handles model updates.
- **Risks:** Vendor lock-in. Price increases. Vendor may sunset product.

**Decision matrix:**

| Factor | Prompt | RAG | Fine-Tune | Vertical SaaS |
|--------|--------|-----|-----------|---------------|
| **Time to deploy** | Hours | 2-4 weeks | 4-12 weeks | 1-2 weeks |
| **Cost per call** | $0.01-1 | $0.20-2 | $0.01-0.1 | Varies (high) |
| **Accuracy upside** | +5-15% | +10-30% | +15-40% | Domain-optimized |
| **Maintenance burden** | Minimal | Moderate | High | None |
| **Lock-in risk** | API vendor | Index tech | Model tech | Vendor SaaS |
| **Best for** | Rapid MVP | Domain specificity + speed | Extreme accuracy needs | Off-the-shelf solutions |

## HARNESS VS RAW MODEL DECISION

When you decide to BUILD (via fine-tuning or custom training), you face a second decision: **do I invest in a multi-agent harness orchestrating multiple models, or do I just use one well-prompted model?**

This is itself a build-or-buy decision. You're not buying a model; you're buying quality through orchestration complexity.

**Raw model:** Single API call to GPT-4, get answer. Cost: $0.05-0.30 per call. Quality: Depends on prompt.

**Multi-agent harness:** Orchestrate 5-10 agents (researcher, analyzer, validator, formatter, etc.). Each agent is a model call. Total cost: $0.50-3 per request. Quality: Higher (each agent specializes, failures are recoverable, validation is built-in).

**When to invest in harness:**

The harness decision is justified only when:

1. **Quality gap is >30%.** If a single model gets 70% accuracy and the harness gets 95%+, the harness pays for itself. If single gets 85% and harness gets 90%, probably not worth it.

2. **Task is naturally decomposable.** The task breaks cleanly into independent steps: research → analyze → validate → format. If the task is monolithic (single step of reasoning), harness overhead is pure waste.

3. **You can afford 10-22x cost increase.** Harness costs roughly 10-22x more than a raw API call. If your cost/user budget is $0.10 and harness costs $1-2, it doesn't fit. If your budget is $10+, it's feasible.

4. **You're willing to maintain it.** Harness = complex orchestration, agent failure paths, debugging multi-agent interactions. This is high-maintenance territory. Only invest if you have dedicated ML/infrastructure engineers.

**Cost-benefit math:**

```
Scenario 1: Quality gap is small
- Raw model: 1 call × $0.10 = $0.10, accuracy 85%
- Harness: 15 calls × $0.10 = $1.50, accuracy 88%
- ROI: 3% accuracy gain for 15x cost = bad trade

Scenario 2: Quality gap is large + high-volume use case
- Raw model: 1 call × $0.10 = $0.10, accuracy 70%
- Harness: 10 calls × $0.10 = $1.00, accuracy 95%
- Volume: 1M calls/month
- Raw cost: $100k/month, 70% accuracy (700k correct)
- Harness cost: $1M/month, 95% accuracy (950k correct)
- ROI: 250k additional correct outputs for $900k extra cost = good trade if each correct output is worth >$3.60
```

**Honest assessment:**

Most teams over-invest in harness. They see "research shows multi-agent improves accuracy" and build a harness for a simple classification task. The accuracy improvement is real (maybe 2-3%), but the cost increase is 10x. The math doesn't work unless your accuracy bar is extremely high and your cost tolerance is extremely high.

If you choose harness, document the quality baseline (single model accuracy) and the improvement threshold (harness must achieve X% accuracy to justify cost). If harness doesn't hit the threshold within 3 months, kill it and go back to raw model.

## MODEL-AGNOSTIC ABSTRACTION LAYER

Whether you choose API, RAG, fine-tuning, or harness, your implementation will depend heavily on a specific model (GPT-4, Claude, Llama, etc.). **But models change every 6-12 months.** A new model may be 50% cheaper, 2x faster, or more capable. You need to be able to swap without rewriting your entire system.

**The abstraction layer is the most important build decision.** It's not a feature; it's your insurance policy against vendor lock-in and your leverage for cost negotiation.

**What the abstraction layer does:**

```
Application Code → Model Abstraction Layer → Actual Model (GPT-4, Claude, Llama, etc.)
```

The layer handles:
- **Prompt formatting:** Each model has different best practices (system prompt syntax, token limits, few-shot examples). Layer normalizes these.
- **Token counting and context windows:** Claude has 200k context, GPT-4 has 128k. Layer accounts for these differences.
- **Retry logic and fallback models:** If Claude is overloaded, fallback to GPT-4. Layer manages this transparently.
- **Cost tracking per model:** Know the per-model cost (GPT-4 costs 10x more than Claude for same task? That's data).
- **A/B testing different models:** Route 10% of requests to a new model, measure quality/cost. Layer enables this.

**Cost of building the abstraction:**

2-4 weeks of engineering, 5-10% latency overhead. Not free, but far cheaper than rebuilding when a new model ships.

**Value:**

- **Can swap models in <4 weeks** instead of 8-12 weeks without abstraction.
- **When Claude-4.5 is released at 50% cost**, you're in market in hours, not months.
- **Vendor negotiation leverage:** You can credibly say "we can swap to competitors in weeks; what's your best price?" and mean it.
- **Hedge against model deprecation:** OpenAI sunsets GPT-3.5; you switch to Claude in parallel without application code changes.

**Abstraction pattern:**

```python
class ModelAbstraction:
    def __init__(self, model_name: str, config: dict):
        self.model = model_name
        self.config = config
        self._model_instance = self._load_model(model_name)

    def call(self, prompt: str, context: dict = None) -> str:
        # Normalize prompt for this model
        formatted_prompt = self._format_prompt(prompt, context)
        # Handle token limits
        truncated = self._truncate_to_context_window(formatted_prompt)
        # Call model
        response = self._model_instance.invoke(truncated)
        # Track cost
        self._log_cost(self.model, tokens_used=len(response))
        return response

    def _format_prompt(self, prompt: str, context: dict) -> str:
        if self.model == "gpt-4":
            return f"<system>{context['system']}</system>\n<user>{prompt}</user>"
        elif self.model == "claude":
            return f"[SYSTEM]\n{context['system']}\n\n[USER]\n{prompt}"
        # ... handle other models
```

**Critical:** The abstraction must be **model-agnostic** — applications code never knows which model is being called. If your code says `if model == "gpt-4": do_special_thing()`, you've defeated the abstraction and rebuilt vendor lock-in.

## REALITY CHECK

- **Failure mode:** Over-building. Teams fine-tune when they should prompt-engineer. Cost is 100x, accuracy gain is 3%.
- **Gate 4 is where most teams fail.** They don't factor maintenance cost into the decision. Two years later they're maintaining a fine-tuned model nobody wants to touch.
- **Latency is often invisible.** Teams skip Gate 3 thinking "we'll optimize later." By then they've committed to RAG and can't ship sub-500ms.
- **Harness is seductive.** Teams see multi-agent papers and build a harness for every task. Harness improves accuracy 2-3%, costs 10-15x more. Only build if the quality gap is >30%.
- **Abstraction layer is non-negotiable.** If you don't build it upfront, you'll rebuild it in 18 months when Claude-5 is released. Better to invest 2-4 weeks early.

**Bonus: Model-Agnostic Abstraction Layer (Reference)**

If you **do** decide to BUILD, abstract the model layer immediately. This protects you when models change (and they will, every 6-12 months).

Design pattern:
```
Application → Abstraction Layer → Model (GPT-4, Claude, Llama, etc.)
```

The abstraction layer handles:
- Prompt formatting (each model has different best-practices)
- Token counting and context window limits
- Retry logic and fallback models
- Cost tracking per model
- A/B testing different models

Cost: 2-4 weeks of engineering, 5-10% latency overhead. Value: Can swap models in 1 hour instead of 1 week. When Claude-4.5 is released and it's 50% cheaper, you switch in hours, not months.

**The Build-or-Buy-or-Fine-Tune Trilemma (Reference)**

You actually have three options, but they create a tension triangle:

- **Build (custom training):** Wins on accuracy for your domain, loses on cost and speed. Requires ML team.
- **Buy (API):** Wins on simplicity and time-to-market, loses on per-request cost at massive scale (>1M requests/month) and customization.
- **Fine-tune:** Promises to balance; actually compromises on all three. Moderate accuracy gain (+10-20%), moderate cost ($50k-500k), moderate complexity (needs ongoing maintenance).

**When to choose each:**
- **Build if:** You have rare, proprietary data + ML team + 18-month horizon
- **Buy if:** API cost per user <$1 + tight latency budget (<200ms) + don't need domain customization
- **Fine-tune if:** You have 10k+ labeled examples + accuracy gap is 15%+ + cost per user is $1-5 + you have maintenance team

In practice: Most products are "Buy" or "Buy + cached prompting." Fine-tune is often the wrong middle ground.

**Model Switching: When and How**

Models change every 6-12 months. You need a plan to switch.

**Decision points:**
- New model is 2x cheaper for same accuracy
- New model is 20%+ faster for same quality
- New model is significantly more capable (enables new features)
- Your current model is deprecated (sunset date)

**Switching cost factors:**
- Prompt reoptimization: 1-4 weeks
- Re-evaluation on your benchmark: 1-2 weeks
- Rollout and monitoring: 1 week
- Total: 3-7 weeks if abstracted, 8-12 weeks if not

If you can't switch in 4 weeks, your abstraction is leaky. Fix it now, not when the vendor forces a switch.

## REALITY CHECK

- **Over-building is the #1 failure mode.** Teams fine-tune to save 20% on tokens, forget the $300k infrastructure cost, and end up paying for years.
- **Gate 5 failure:** Teams skip maintenance assessment. Two years later they're maintaining an unmaintained fine-tune.
- **Gates are not weighted.** A "no" on any gate means "buy." You can't trade off "no maintenance team" for "high cost tolerance."
- **Cost per user <$1 = wrong decision.** If API cost per user is low, the economics almost never support building.
- **Model switching is not optional.** The average product switches models every 12-18 months. Expect it. Plan for it.

## WHEN WRONG

- Very early exploration (just ship, don't optimize yet)
- High-touch, single-customer (cost/user math breaks down)
- When a gate's assumption changes (re-run the framework)
- When building is the learning goal (R&D, not product)
