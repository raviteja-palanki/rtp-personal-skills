## DEPTH DECISION

**Go deep if:** You're designing multi-agent systems at scale (10+), shipping to production with legal/financial liability, or implementing progressive trust escalation across diverse action types.

**Skim to diagnostic questions if:** You need a quick permission model for a single agent, or want to audit existing autonomy thresholds.

**Skip if:** Pure advisory systems (read-only, no actions), regulatory approval already in place, or zero autonomy needed.

## THE TRAP

You reverse-engineer permissions from capability: "agent CAN read files, so let it." The trap is missing failure cost. Start with cost instead: "if agent reads wrong files, what's the consequence magnitude?" Autonomy is not binary. Neither is trust. Conflating them kills both.

## THE PROCESS

**Layer 1: Classify the action**

- Reversible, low-consequence (read observation, cost <$1): → ADVISOR (recommend, no gates)
- Reversible, medium-consequence (modify-in-sandbox, cost <$100): → CO-PILOT (ask, but can execute if approved)
- Irreversible, medium-consequence (publish, cost $100-10k): → LOCKED (requires human signature)
- Irreversible, high-consequence (delete production, cost >$10k or data loss): → REFUSE (agent cannot access, period)

**Layer 2: Map confidence to autonomy gates**

For each action class, define:
- Confidence threshold to trigger autonomous execution (if >X% certain, act; else ask)
- Approval routing (human, policy rules, time-based gates)
- Audit logging before and after execution

Example chain:
- Agent confidence <60%: always ask
- Agent confidence 60-85%: ask if cost >$500
- Agent confidence >85%: execute if reversible, ask if medium-consequence
- Agent confidence 95%+: execute all non-locked actions (trust earned)

**Layer 3: Earn trust progressively**

- Track agent decision quality (did it predict correctly, did side effects match forecast?)
- Adjust autonomy ceiling based on streak length (3 correct predictions → raise threshold 5%)
- Reset streak on error; never auto-raise beyond domain boundary
- Expose progress to user (transparency prevents surprise autonomy escalation)

**Layer 4: Permission scoping**

- Bind autonomy to time window (auto-approval expires after 24h)
- Bind to actor context (different autonomy in "batch job" vs "interactive user mode")
- Bind to resource class (full autonomy over dev-only, read-only over prod)

**Layer 5: HITL Patterns and Leash Length**

Autonomy isn't binary. Design the "leash length" — how much rope to give at each trust level:

- **Supervised mode (leash = 0):** Human watches every decision in real-time. Agent proposes, human approves synchronously. Latency cost: 100-500ms per action. Use when: learning agent behavior, catastrophic consequence magnitude possible.

- **Spot-check mode (leash = task):** Agent acts freely within a task boundary. Humans audit decisions post-execution within a window (5 minutes, 1 hour). Use when: reversible actions, human can intervene before effects propagate.

- **Exception-based mode (leash = domain):** Agent acts freely; escalate only if confidence drops below threshold or error detected. Use when: high accuracy domain, failure costs predictable.

- **Autonomous mode (leash = unlimited):** Agent decides and executes without gates. Humans monitor only via dashboards. Use when: >99% accuracy, consequence magnitude <$10, or read-only operations.

Each level requires different infrastructure. Don't skip it in design.

**Layer 6: Rollback Mechanisms**

Autonomy without undo is reckless. Design three levels of rollback:

1. **Immediate rollback (5-minute window):** Human can undo action within 5 minutes. Requires transaction log + state snapshot. Example: revert file modification, undo send.

2. **Audit-and-restore (24-hour window):** Action happened; audit shows who authorized. Restore requires manual review but is possible. Example: revert database write, restore deleted file from backup.

3. **No rollback (permanent):** Action is irreversible. Only option: compensate (refund, apologize, re-execute correctly). Example: sent email, published article, deleted without backup.

Never grant autonomy to "no rollback" actions without human signature. For each action class, define rollback window explicitly. If no rollback exists, raise confidence threshold by 10-20% or reduce consequence magnitude instead.

```
## Autonomy Spec: [Feature/Agent Name]

**Capability Domain:** [e.g., file operations, user messaging, analytics export]
**Typical Context:** [e.g., batch job, real-time user interaction, background task]
**Owner:** [Team responsible for oversight]

### Action Inventory

| Action | Type | Reversibility | Consequence Magnitude | Base Autonomy Level | Confidence Threshold |
|--------|------|--------------|--------------|-------------------|---------------------|
| [Action A] | [read/write/delete] | [immediate/24h/none] | [$amount or data loss] | [1-4] | [50-95%] |
| [Action B] | [read/write/delete] | [immediate/24h/none] | [$amount or data loss] | [1-4] | [50-95%] |

### Leash Configuration

| Trust State | Leash Length | Approval Path | Rollback Window | Context Limit |
|-------------|-------------|---------------|-----------------|---------------|
| Bootstrapping (0-3 decisions) | task-level | sync approval, ops team | 24h | 30% |
| Building (4-20 decisions) | domain-level | async queue, threshold-based | 24h | 50% |
| Established (20+ decisions, <2% error) | unlimited | exception-based escalation | 5-min for reversible | 60% |
| Mature (100+ decisions, <0.5% error) | org-wide | monitoring dashboards | 5-min for reversible | 70% |

### Progressive Trust Rules

- **Raise autonomy after:** 3+ consecutive correct decisions within 72-hour window
- **Reset streak on:** Any decision that required post-hoc correction, confidence miscalibration, or side effects not forecasted
- **Max autonomy ceiling:** Level [3 for reversible, 2 for medium-consequence, 1 for irreversible, 0 for locked]
- **Review cadence:** Weekly for <10 decisions/week; daily for >50 decisions/week
- **Confidence recalibration:** Monthly validation against actual error rates
- **Context-gated rules:** Autonomy drops 1 level if context utilization >50%, advisor-only if >70%

### Decision History & Feedback Loop

- Audit all decisions with: agent ID, timestamp, action, confidence, approved_by (if applicable), outcome, duration_to_rollback
- Track confidence calibration: Does agent confidence >80% predict 80% accuracy? If not, recalibrate thresholds
- Monthly audit of escalations: Why were they escalated? Would higher confidence have been safe?
- Quarterly stakeholder review: Is autonomy level still aligned with business risk tolerance?
```

This template ensures autonomy specs are actionable, auditable, and transparent to stakeholders.

**Layer 7: HITL Decision Routing**

When an action requires human input, where does it go?

- **Synchronous escalation:** Agent pauses, waits for approval. Adds latency; unscalable at 1000s of requests.
- **Async approval queue:** Agent enqueues decision, continues other work. Human approves from inbox. Breaks user expectations (why wait if agent already acts?).
- **Policy-automated approval:** Certain patterns (e.g., confidence >90%, cost <$100, reversible) pre-approved. Reduces human bottleneck. Requires careful audit.
- **Round-robin assignment:** Escalations distributed to available humans. Fair load, but slow.

Design routing per action type, not per agent. Different actions have different escalation paths.

## THE CONTEXT ANXIETY PRINCIPLE

**Context Anxiety in Autonomous Agents** — Anthropic March 2026 finding: as context fills, agents wrap up work prematurely. They produce "good enough" outputs instead of excellent ones because they sense the context window closing. Impact on autonomy design: don't grant high autonomy to agents near context limits. Monitor context utilization as a signal — quality degrades at 50-60% of max context (the Pre-Rot Threshold), not at 100%. Solution: context resets between decisions, file-based state persistence, and hard limits on context per autonomy decision.

**Why this matters for autonomy:**
- Agents at 70% context utilization began making hasty autonomous decisions (confidence 85%+) that would have required escalation at 30% utilization
- The agent wasn't less capable; it was anxious. It picked speed over accuracy
- Autonomy thresholds tuned at comfortable context levels will fail at high utilization
- This is invisible in testing (test context is always fresh)

**Implementation:**
1. Never grant autonomy Level 3+ when agent context > 50% of max
2. Reset context before autonomy decisions, or reduce autonomy ceiling automatically
3. Log context % at each autonomy decision; correlate with error rate
4. If error rate spikes when context >60%, lower autonomy ceilings or add context resets

**Signal detection:**
- Agent autonomously executes decision that barely meets confidence threshold
- Same decision would have escalated 10 minutes earlier in the session
- Agent's confidence estimates become sticky (don't fluctuate as much)
- Agent stops asking clarifying questions and starts assuming

**Leash design for context-aware autonomy:**
- At 0-40% context: Normal autonomy rules apply
- At 40-60% context (Pre-Rot): Drop autonomy by 1 level, surface uncertain decisions, require sync approval for irreversible actions
- At 60%+ context: Advisor mode only (recommend, never execute autonomous)
- At 80%+ context: Read-only mode (no new decisions, conclude current task)

This shifts the mental model: autonomy is not constant. It's dynamic, gated by context state.

## REALITY CHECK

**Failure modes:**

1. Autonomy creep: You raise thresholds, forget to lower them on errors. Now your agent has too much power and no one notices until a consequence event happens.
   → FIX: Set explicit timeout/expiry on autonomy grants. Force audit every 30 days.

2. Silent escalation: Agent hits 95% confidence on task A, so you raise overall autonomy. But task B has different error distribution. Agent is now overconfident on B.
   → FIX: Autonomy is task-specific, not agent-global. Use separate tracks per domain.

3. User learned helplessness: Agent asks permission on 200 actions, user always clicks yes. Now user is a rubber stamp.
   → FIX: Bundle decisions (ask once per batch, not once per item). Surface only decisions where agent is uncertain.

4. Audit trail gaps: Autonomous execution with no logging = post-hoc liability. "The agent did what?"
   → FIX: Log decision metadata (confidence, gates applied, approval route). Searchable by outcome.

**Cost at 10x scale:**
- 1 agent, 1k decisions: autonomy saves time, minor consequence magnitude
- 10 agents, 10k decisions: cascading failures at low autonomy threshold, bottleneck at high threshold
- 100 agents, 100k decisions: trust infrastructure becomes the product (you're building governance, not agent)

**Latency impact:**
- Advisor mode: +0ms (parallel execution with user)
- Co-pilot mode: +100-500ms (fetch approval route, wait for signature)
- Locked mode: +overhead of delegation (explain to human, wait for escalation)

## QUALITY GATE

- [ ] Every action is classified by reversibility and consequence magnitude (not by capability)
- [ ] Confidence thresholds are empirically tied to past error rates, not intuition
- [ ] Autonomy raises only after 3+ consecutive correct predictions; resets on error
- [ ] Users see autonomy level and can adjust it (transparency, not hidden governance)
- [ ] Audit log is queryable by (agent, action, outcome, confidence, approval_route)

## WHEN WRONG

This skill gives bad advice if:

1. Pure-advisory domain (read-only, zero blast). Just let agent analyze freely.
2. Error distribution non-stationary. Confidence doesn't predict accuracy. Audit first.
3. Users intolerant of wait time. Async approval breaks UX. Reduce consequence magnitude instead.
4. No reversibility boundary. Every action permanent. Plan undo or use failure-design.
5. Trust is organizational (regulatory sign-off). Use determinism-compass for policy alignment.
