# Strategy Canvas

## DEPTH DECISION

**Go deep if:** You're setting 12-month strategy for AI product, evaluating roadmap for model capability changes, or designing for moat defensibility in a fast-moving space.

**Skim to diagnostic questions if:** You want a quick strategy audit or quarterly refresh.

**Skip if:** Pre-product-market fit where experimentation matters more, or if you already have a validated capability-conditional strategy.

## GROUNDING (Before Starting)

Follow the [Universal Skill Protocol](../../../UNIVERSAL-SKILL-PROTOCOL.md):
1. Ask the Grounding Questions (Section 1) — at minimum: Who is the customer? What problem? What are we saying YES to and NO to?
2. Route depth: Executive Summary or Comprehensive Analysis?
3. Identify output format: Document, presentation, spreadsheet, or inline?

Then proceed with the skill-specific analysis below.

## THE TRAP

You will write a static strategy and call it "adaptive." The bias is **planning fallacy** — you'll design a 12-month roadmap against today's model capability, and three months later a new model breaks it. The bias compounds because static strategies *look* professional (printed roadmaps, quarterly milestones) while adaptive strategies look messy (conditional bets, regular re-evaluation). You'll optimize for the wrong thing.

AI strategy is not traditional product strategy on a faster clock. The capability frontier moves under your feet quarterly. A strategy that predicts this becomes fiction. A strategy that adapts to it becomes defensible.

## THE PROCESS

**Step 1: Separate stable from volatile.** Create a table:
- **Stable (12+ mo):** User problems, market structure, regulatory, org capabilities
- **Volatile (3-6 mo):** Model capability, token cost, competitor feature parity, tool availability

Strategy anchors on stable. Strategy adapts on volatile.

**Step 2: Define capability-conditional bets.** For each major product bet, write:
"IF [capability threshold] by [date], THEN build [path A]. IF NOT, THEN build [path B]."

Example: "IF multi-step reasoning reaches >90% accuracy by Q2, THEN invest in autonomous agents. IF NOT, THEN invest in AI-assisted human agents."

Each bet has trigger (measurable), condition (external), and fallback (ready to execute).

**Step 3: Identify unique context advantage.** What can't a competitor replicate by switching models?
- Proprietary data
- Workflow depth in a specific domain
- Enterprise trust/compliance credentials
- Customer integration lock-in

Pick 1-2. Moat without multiple vectors is fragile.

**Step 4: Build one-page strategy canvas:**
```
PROBLEM      | What user problems are we solving? (stable)
CAPABILITY-BETS | Path A (IF X), Path B (IF NOT X)
MOAT         | Which defensible advantage(s)?
TRIGGERS     | When do we pivot? Quarterly review dates.
```

**Step 5: Account for capability shift problem.** Features you build, foundation models absorb.

Example: You ship "context window optimization" as a feature (compress docs into summaries). In 6 months, base models include 200k context windows. Your feature is obsolete. This isn't failure; it's the frontier moving.

**Defense strategies:**
- **Deepening:** Instead of "optimization," shift to "judgment" (which docs matter). Requires proprietary training data.
- **Lock-in:** Embed your feature in workflow so deep that switching costs exceed the value of model improvements (integration moat).
- **Velocity:** Ship faster than models improve. Not sustainable long-term, but buys time.

For each feature, estimate: "If the model absorbs this capability, what's our fallback?" If no fallback exists, that feature is a risk, not a moat.

**Step 5.5: Estimate strategy half-life.** How long before model improvements invalidate your strategy?

- **0-3 months:** Frontier features (long-context, tool use, reasoning). Strategy based on these is dead on arrival.
- **3-6 months:** Competitive features (speed, cost, quality). Expect to pivot twice.
- **6-12 months:** Market structure. Slower to shift but not immune.
- **12+ months:** User problems and organizational capabilities. Most stable.

Build strategy primarily on 6-12 month anchors. Use 3-6 month contingencies. Ignore 0-3 month frontier volatility (except as triggers, not bets).

**Step 5.75: Agentic PMF lens.** If your product will be agent-first (autonomous, multi-step), adjust your strategy.

Traditional product: user asks → AI answers. PMF = accuracy + latency + price.
Agentic product: user wants outcome → agent plans steps → executes. PMF = reliability + error recovery + autonomy level.

**Capability bets shift:**
- NOT: "If LLM reaches 95% accuracy, build X"
- BUT: "If multi-step agentic reasoning reaches 85% success rate with <2 hallucinations per 10 steps, build autonomous product. Otherwise, co-pilot mode."

Agentic strategy requires **failure modes** to be explicit in your bet (you won't hit 100% accuracy; the question is: can users trust 85%?). Traditional strategy optimizes for best-case. Agentic strategy optimizes for worst-case recovery.

## Harness Strategy

Your harness (agent orchestration, context management, eval pipeline) is a strategic asset that survives model upgrades. This is separate from model capability bets.

**What is the harness?**
- Orchestration: How agents coordinate multiple steps, tools, and decisions
- Context management: How you assemble, cache, and refresh context
- Eval infrastructure: How you measure agent quality systematically
- Tool integration: How deeply agents are embedded in your workflows
- Error recovery: How agents handle failure and retry

**Why it matters strategically:**
When the next model drops (o4, Grok v3, whatever comes next), you swap the model and keep the harness. Teams that invest in harness architecture as a strategic asset:
- Can upgrade models with one-line config changes
- Don't get locked into single vendor or provider
- Can run multi-model strategies (cheap model for routine tasks, expensive model for complex reasoning)

**Teams that ignore harness architecture:**
- Rewrite their entire product when moving models
- Get locked into single provider (can't leave due to rewrite cost)
- Are vulnerable to pricing changes or provider deprecation

**Action:** Architect your harness as a strategic bet independent from model choice. Document harness decisions (orchestration framework, context caching strategy, eval metrics) as architectural decisions, not temporary implementation choices.

## The Model-Agnostic Moat

The moat is not the model. The moat is what you can do with the model that competitors can't replicate by switching models.

**Model-dependent (fragile moat):**
- "We built X using GPT-4's capability"
- Competitor responds: "We built X better using Claude 3.5"
- Your moat evaporates

**Model-agnostic (durable moat):**
- "We have an abstraction layer so we swap models without rewriting product"
- Competitor: "We'll build the same"
- But you've got a 6-month head start and customer lock-in through that abstraction

**How to build model-agnostic moat:**
1. Create an abstraction layer between product and model (LangChain-style, or custom)
2. Document model swap costs (how many lines of code must change? None? 10 lines? 100?)
3. Plan model rotation: quarterly evaluate if cheaper/faster/better models exist, swap if justified
4. Lock-in through workflow depth: even if competitors build the same abstraction, your customers are integrated into a workflow they don't want to leave

**Example:**
- Fragile: "We built a medical assistant using GPT-4's reasoning" (competitor builds better with Claude, you lose)
- Durable: "We have a medical assistant with a model-agnostic harness and 10 domain-specific tools integrated into hospital workflows" (competitor can swap models, but integrating into hospital workflows takes 6 months; customers are locked in)

**Action:** Separate architectural decisions (abstraction layer, harness design, tool framework) from model-dependent decisions (which model to use, which capability to rely on). Invest heavily in architecture.

## Strategy for Agentic Products (Expanded)

Agentic products have fundamentally different strategic dynamics because the product IS the harness.

**Traditional AI product strategy:** Features matter. Speed matters. Cost matters. Model capability matters.

**Agentic product strategy:** Harness maturity matters most. What matters:
- **Eval infrastructure:** Can you measure agent success systematically across thousands of scenarios?
- **Tool integration depth:** How tightly integrated are agents in customer workflows?
- **Error recovery:** How well do agents recover from failures without human intervention?
- **Multi-step reasoning reliability:** How consistently does your orchestration work?

**Strategic implications:**
- Don't chase model upgrades. Instead, invest 40% in eval infrastructure, 30% in tool integration, 20% in error recovery, 10% in model updates.
- Your competitive moat is eval infrastructure + orchestration quality, not model quality.
- Plan for agents to fail sometimes. Your moat is failing gracefully.

**Capability bets for agentic products:**
- NOT: "If LLM reaches 95% accuracy, ship autonomous agents"
- BUT: "If we can measure agent success across 1000+ scenarios, and error recovery works 95% of the time, ship autonomous agents. We accept 80% accuracy on individual steps."

**Failure mode planning:**
- Traditional product: "If model accuracy drops 5%, users notice immediately"
- Agentic product: "If multi-step reliability drops 10%, users notice eventually. Invest in monitoring and error recovery so failures are transparent."

**Action:** For agentic products, reframe strategy around harness maturity and tool integration depth, not model capability. Set quarterly eval infrastructure goals, not accuracy goals.

**STABLE ANCHORS** (valid 12+ months):

- [Market structure]
- [Regulatory constraint]
- [Org capability]

**VOLATILE ASSUMPTIONS** (valid 3-6 months):
- [Model capability]
- [Token cost]
- [Competitor feature parity]

**CAPABILITY-CONDITIONAL BETS:**
1. IF [trigger] by [date] THEN [Path A]. ELSE [Path B].
2. IF [trigger] by [date] THEN [Path A]. ELSE [Path B].
3. IF [trigger] by [date] THEN [Path A]. ELSE [Path B].

**MOAT:**
- Type: [proprietary data | workflow depth | tool integration | enterprise trust]
- Evidence: [specific advantage vs. competitor]
- Durability: [survives model change? Y/N]

**HARNESS STRATEGY:**
- What survives model upgrades: [specific harness components]
- Model swap cost: [# lines changed, or "config only"]
- Tool integration depth: [current level, target level]

**AGENTIC PRODUCT NOTES** (if applicable):
- Eval infrastructure maturity: [current | target]
- Error recovery strategy: [how agents handle failure]
- Multi-step reliability target: [% success rate]

**STRATEGY HALF-LIFE:** [3 months | 6 months | 12 months]

**NEXT REVIEW:** [specific date, typically quarterly]

**Step 6: Set quarterly cadence.** Schedule reviews. Test assumptions:
- Did model capability hit trigger?
- Did cost structure change?
- Did competitor move?
- Do we still have the moat?
- Did capability shift eat any of our features?
- Do we need to deepen, lock-in, or accelerate?

## REALITY CHECK

- **Prediction fails.** You will be wrong about model capability timelines. Design for scenario planning, not point estimates.
- **"Use the best model" is not strategy.** It's a dependency. Real strategy is: what do we do when the best model is the competitor's?
- **Strategy without moat is a feature list.** It will be copied in weeks. Real strategy answers: why can only we solve this?
- **Quarterly reviews must happen.** If you skip them, this framework becomes static theater. Schedule them now; they're non-negotiable.
- **Harness survives models.** The strategic asset is the orchestration, context management, and tool integration—not the model. Build accordingly.

## QUALITY GATE

- [ ] Stable vs volatile assumptions separated
- [ ] Strategic bets are capability-conditional with explicit triggers
- [ ] Unique context advantage identified and specific
- [ ] Strategy canvas produced as a one-page artifact
- [ ] Quarterly review cadence set with specific trigger-check protocol
- [ ] Moat is model-agnostic (survives model swaps)
- [ ] Harness strategy explicitly documented
- [ ] If agentic product: eval infrastructure and error recovery strategies included

## WHEN WRONG

- Very early-stage companies where finding product-market fit matters more than strategy
- When the team needs to execute, not strategize (analysis paralysis risk)
- When the market is so new that strategic analysis has insufficient data
