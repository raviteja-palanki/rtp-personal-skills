# Failure Design

## DEPTH DECISION

**Go deep if:** Designing failure UX for a customer-facing feature or a high-stakes decision point. **Skim to questions if:** Reviewing fallback handling in a feature spec. **Skip if:** Internal tools where failure cost is low and users verify all outputs.

## GROUNDING (Before Starting)

Follow the [Universal Skill Protocol](../../../../UNIVERSAL-SKILL-PROTOCOL.md):
1. Ask the Grounding Questions (Section 1) — at minimum: Who is the customer? What problem? What are we saying YES to and NO to?
2. Route depth: Executive Summary or Comprehensive Analysis?
3. Identify output format: Document, presentation, spreadsheet, or inline?

Then proceed with the skill-specific analysis below.

## THE TRAP

You will design for the happy path. The bias is **optimism bias** — you'll spec the success case and create a one-line item in the PRD labeled "error handling." But in AI products, the AI being wrong isn't an edge case — it's a guaranteed operating condition. Designing for failure isn't defensive engineering. It's the core UX.

The worst variant: designing error states that blame the user. "Try rephrasing your question" is the AI equivalent of "User error." The system failed, but the response implies the user asked wrong. Compare: "I'm not confident in this answer. Here's what I found, but verify before acting." Same failure, different trust trajectory. One erodes trust; the other builds it.

## THE PROCESS

1. **List every AI output in the feature.** Not "the model responds." Specifically: what generated text, classification, recommendation, or action does the user see or experience?

2. **For each output, answer: "What happens when this is wrong?"**
   - Wrong and user notices immediately → Design correction path
   - Wrong and user notices later → Design recovery path + trust repair
   - Wrong and user never notices → Design verification mechanism
   - Wrong and causes downstream action → Design undo/reversal mechanism

3. **Design the failure UX for each scenario:**
   - **Transparency:** Does the user know AI generated this? Confidence indicators?
   - **Correction:** Can the user fix it inline without starting over?
   - **Escalation:** Can the user reach a human/alternative path?
   - **Learning:** Does the correction feed back into the system?

4. **Define the refusal boundary.** When should the AI say "I don't know" or "I can't do this safely" instead of guessing?
   - What confidence threshold triggers refusal?
   - What does the refusal look like to the user?
   - Is refusal better than a low-confidence answer? (Usually yes.)

5. **Write failure acceptance criteria.** For every user story, add:
   - "When the AI is wrong, the user can [specific correction action]"
   - "When the AI is uncertain, the system [specific fallback behavior]"
   - "When the AI fails completely, the user sees [specific degraded experience]"

6. **Define graceful degradation hierarchy.** When AI fails, what's the fallback chain?
   - **Cache:** Return last known good result (fast, stale)
   - **Simpler model:** Swap to cheaper, more reliable model (slower, less capable)
   - **Rules:** Switch to deterministic fallback (no surprise failures)
   - **Human escalation:** Hand off to customer support (slow, high trust)
   - **Error state:** Show degraded experience or "try later" (worst case, last resort)
   - Each step trades speed for reliability. Define thresholds for when to cascade down.

## FAILURE TAXONOMY FOR AI PRODUCTS

| Failure Type | User Experience | Detection | Recovery Pattern |
|-------------|----------------|-----------|-----------------|
| **Hallucination** | Plausible but wrong answer | Hard — user may not notice | Citation verification, confidence threshold, encourage user verification |
| **Refusal** | "I can't help with that" | Easy — explicit | Explain why, offer alternative path or escalation |
| **Latency timeout** | Spinner/loading forever | Easy — timer | Progressive display, cancel option, fallback to cached result |
| **Capacity error** | Service unavailable | Easy — error code | Queue, retry, cache fallback, degraded mode |
| **Safety trigger** | Content blocked/filtered | Medium — may over-block | Appeal mechanism, human review, confidence override |
| **Partial failure** | Incomplete or truncated output | Medium | Detect truncation, offer regeneration, show partial results |
| **Cascade failure** | Multi-agent chain breaks | Hard — failure propagates | Circuit breaker, isolate components, rollback to last checkpoint |
| **Silent degradation** | Quality drops without error | Hardest — no signal | Continuous monitoring, eval pipeline, confidence decay alerts |

## GRACEFUL DEGRADATION HIERARCHY (WORKED EXAMPLE)

**Feature: AI customer support agent**

- **Level 1 (Cache):** Return cached answer for known FAQ questions
  - Trigger: Exact phrase match on 50+ historical similar questions
  - Latency: 50ms, Quality: High (proven), Cost: Near-zero (pre-computed)
  - Covers: 85% of incoming queries
  - User experience: Instant response with "This is from our knowledge base" indicator

- **Level 2 (Simpler model):** Route to smaller/faster model for simple queries
  - Trigger: Low complexity score (single topic, no multi-step reasoning required)
  - Latency: 200ms, Quality: 78%, Cost: Low (cheaper inference)
  - Covers: 10% of queries (routing from Level 1 miss)
  - User experience: Normal latency, note "powered by quick lookup"

- **Level 3 (Rules):** Pattern-match known issue types to templated responses
  - Trigger: Confidence < 60% after Level 2 attempt
  - Latency: 10ms, Quality: 85% (deterministic), Cost: Minimal
  - Covers: 4% of queries (edge cases, specific issue categories)
  - User experience: Human-like response, includes escalation link

- **Level 4 (Human):** Create support ticket with AI-gathered context pre-filled
  - Trigger: All previous levels below acceptance threshold
  - Latency: Async (agent queued), Quality: 99%, Cost: Human labor
  - Covers: 1% of complex/urgent queries
  - User experience: "I'm creating a support ticket for you. Here's what I found:" + ticket link

- **Level 5 (Error):** Explicit degraded state when system is unhealthy
  - Trigger: Service unavailable, cascade failure detected, rate limit hit
  - Latency: Instant, Quality: Honest failure, Cost: Reputation (high)
  - Covers: Critical outages only
  - User experience: "I'm having trouble right now. Here's how to reach our team by phone/email" + direct contact info

**Each level includes:** trigger condition (measured), latency impact (vs Level 1), quality trade-off (acceptance % or user rating), cost impact (engineering effort, compute, human time).

## FAILURE UX PATTERNS

- **Progressive confidence:** Show confidence inline, not as a separate warning. "I'm fairly confident this is the answer, but verify the date" vs generic "AI-generated content may be inaccurate." Let confidence match the context of use.

- **Inline correction:** Let users edit AI output in place without starting over. Example: User highlights a word in the AI response and clicks to edit. Changes apply instantly, no "regenerate from scratch" button needed. Trains the AI on the correction if approved.

- **Explanation on failure:** "I couldn't find recent pricing for Product X because our knowledge base was last updated in January" is better than "No results found." Users understand the constraint and can decide: try a different question, or escalate to human.

- **Undo/revert:** Every AI action should have a single-click undo for at least 30 seconds. Applied filters, suggested edits, auto-complete insertions—all revertible. After undo window closes, require confirmation for any repeat action.

- **Feedback on failure:** Make it dead easy to report bad output. Thumbs down, flag for review, or short comment field. Route high-volume bad outputs to a queue for model improvement.

- **Degraded mode indicator:** Show users explicitly when they're getting a fallback experience. "You're offline—showing cached results from 2 hours ago" or "High demand—using faster model, quality may vary." Transparency builds trust even in failure.

## FAILURE COST MATRIX

| Failure Scenario | Low Stakes (autocomplete) | Medium Stakes (search results) | High Stakes (medical/financial recommendations) |
|---------|--------------------------|----------------------|-------------------------------|
| **Design investment** | Dismiss button or undo-within-5s | Correction + show alternatives | Full verification + mandatory human review |
| **Detection speed** | Batch eval (daily or weekly) | Near real-time (per user session) | Real-time (per response, before display) |
| **Recovery path** | Auto-dismiss after 5s, user can undo | Offer 3-5 alternatives, escalation link | Block action, require human approval to proceed |
| **Cost of building** | 1-2 days engineering | 1-2 weeks engineering | 1-2 months engineering (including safety review) |
| **Trust damage if deployed wrong** | Low (context forgiven) | Medium (user frustration) | Critical (regulatory, legal, reputation) |
| **Acceptance criteria** | "Wrong 15% of time is fine" | "Wrong 5% or less, with alternatives shown" | "Wrong < 0.1%, human verification required" |

## DIAGNOSTIC QUESTIONS

Ask yourself these before designing failure UX:

- **"For each AI output, what's the cost of being wrong?"** If I can't quantify it in dollars, reputation, time, or safety impact, I can't design the right failure UX. Autocomplete wrong = seconds to undo. Medical recommendation wrong = patient harm. Design proportionally.

- **"Is the user aware this output is AI-generated?"** If not, failure trust damage is worse. A human writing the same wrong answer earns benefit of doubt. An AI writing the same answer triggers "I knew AI makes stuff up." Build transparency early, not after failure.

- **"What's the time between failure and user discovery?"** Minutes (user reviews immediately) = manageable. Hours (email goes out, user reads later) = bad but fixable. Days (compounding effect occurs) = catastrophic. Design detection speed proportionally.

- **"Does the failure compound?"** AI recommends wrong stock → user buys → price drops → user loses money AND trust erodes. vs AI autocompletes wrong word → user clicks delete. Compounding failures need prevention, not just recovery.

- **"What's the recovery cost for the user?"** Seconds to undo vs hours to fix downstream damage. If recovery is expensive, prevention is cheaper. If recovery is free, lighter-weight UX is acceptable.

### Recovery Cost Spectrum

Not all failures cost the same to recover from. The spectrum runs from "under a minute" to "never fully recoverable":

**Tier 1 — Seconds (Undo):**
- Text generation wrong → user deletes, tries again. Cost: 5 seconds.
- Suggested action declined → no action taken. Cost: 0.
- Wrong autocomplete → user corrects. Cost: 2 seconds.

**Tier 2 — Minutes to Hours (Repair):**
- Email sent with wrong content → apologize, send correction. Cost: user trust, time.
- Calendar event created incorrectly → delete, recreate. Cost: 10 minutes.
- Code suggested and committed with a bug → debug, fix, redeploy. Cost: 1-4 hours.

**Tier 3 — Days to Weeks (Remediation):**
- Bulk records incorrectly updated → audit, identify affected rows, correct. Cost: 1-2 weeks.
- Wrong financial advice acted on → reverse transactions, notify affected parties. Cost: weeks, legal risk.
- Data pipeline generating wrong labels for 2 weeks → retrain model on corrected data. Cost: 2-4 weeks.

**Tier 4 — Permanent (Irrecoverable):**
- Personally identifiable data leaked → breach notification required, reputational damage. Can't take it back.
- Safety-critical instruction followed by user → health consequences. No remediation.
- Customer churned because of repeated AI failures → maybe they never come back.

### The Time-to-Recover Dimension

Recovery cost has two dimensions: **effort** (engineering hours, money) and **time** (calendar days before normal service resumes). These move independently.

| Failure Type | Effort to Recover | Time to Recover |
|---|---|---|
| Wrong AI suggestion (user declined) | None | Instant |
| Wrong AI suggestion (user acted on) | Low-Medium | Minutes-Hours |
| Corrupted data from AI action | High | Days-Weeks |
| Reputational failure (public incident) | Very High | Months-Years |
| Regulatory/legal consequence | Extreme | Ongoing |

**Design implication by user type:**
- **Power user (expert, daily use):** Tolerates higher false positive rate. Recovers from errors quickly (knows what's wrong, knows how to fix). Design for speed.
- **Casual user (occasional, low expertise):** Doesn't know when the AI is wrong. Errors compound because they trust the output. Design for safety — prefer "I'm not sure" over a confident wrong answer.
- **Enterprise admin (high-stakes, compliance):** Wrong action might affect thousands of downstream records. Recovery cost is extreme. Design for confirmation dialogs, audit logs, and reversibility.

- **"What's my refusal threshold?"** Where does "I don't know" live? 60% confidence? 50%? Model it: If users ignore low-confidence answers, raise threshold. If threshold too high, system becomes useless. Measure and tune empirically.

## MULTI-AGENT FAILURE DESIGN

When multiple AI agents hand off work to each other:

- **Circuit breaker pattern:** If agent fails 3 times in 10 minutes (measured by downstream user feedback), route future requests to fallback path. Reset circuit breaker when agent succeeds 5 times. Prevents cascade of bad outputs.

- **Isolation:** Each agent's failure shouldn't cascade to others. If Agent A (summarizer) fails, Agent B (analyzer) still works on raw input. Design handoff contracts: if A can't deliver, B has a fallback input type.

- **Checkpoint/rollback:** Save system state before each agent hand-off. If Agent B fails, revert to Agent A's output and try Agent C instead. Users see one result, not a broken chain.

- **Timeout per agent step, not just overall:** Agent A has 2s, Agent B has 3s, Agent C has 1s. If any times out, don't wait for the chain—escalate to fallback. Total timeout is 6s, not "wait forever."

- **Error attribution:** Which agent in the chain caused the failure? Log: Agent → Operation → Latency → Failure Type → Recovery Path. Use this signal to optimize: swap Agent D for Agent A, adjust Agent B's timeout, add circuit breaker before Agent C.

## REALITY CHECK

- **Over-warning trap:** If every AI output comes with a disclaimer, users develop warning fatigue and ignore them all. Transparency is not "more warnings" — it's calibrated transparency matched to risk. Low-risk outputs (grammar suggestions) need minimal transparency. High-risk outputs (medical recommendations) need explicit confidence and reasoning.
- **Correction cost vs consequence magnitude:** Every correction path is engineering work. Prioritize by cost of failure: high-stakes outputs (financial recommendations, medical decisions) get rich correction and escalation UX. Low-stakes outputs (autocomplete suggestions) can use simple dismiss/undo.
- **The refusal paradox:** A system that refuses too much is useless (users get frustrated). A system that never refuses is dangerous (users trust unreliable output). The threshold is a product and business decision, not a technical one. It should be measurable and tunable.
- **Confidence theater:** Showing confidence percentages that users can't act on is worse than showing nothing. "87% confidence" is meaningless. "I'm fairly confident in this" or "You should verify this" is actionable.

## QUALITY GATE

- [ ] Every AI output in the feature has a defined "when wrong" scenario
- [ ] Correction, recovery, or verification path designed and costed for each
- [ ] Refusal boundary defined with specific confidence threshold and reasoning
- [ ] Failure acceptance criteria written for every user story ("when AI is wrong, user can...")
- [ ] Failure UX reviewed by someone outside the design team (fresh perspective)
- [ ] At least one failure scenario prototyped or tested with users
- [ ] Failure cost matrix completed and prioritization defense clear
- [ ] Degradation hierarchy mapped with trigger conditions and SLAs
- [ ] Multi-agent failure design documented (if applicable)
- [ ] Diagnostic questions answered for each high-stakes output

## WHEN WRONG

- Internal tools where failure is cheap and users are trained to verify outputs
- Experimental/beta features where users have explicitly opted into lower quality
- For deterministic components (rules-based error handling is different)
- When failure design becomes a delay tactic instead of a shipping path (use it to move faster, not slower)
